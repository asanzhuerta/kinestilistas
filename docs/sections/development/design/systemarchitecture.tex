\subsection{Diseño de alto nivel}

KinEstilistas se implementa como una aplicación web monolitica full-stack. La misma base de código contiene tanto la interfaz de usuario como la lógica de servidor y el acceso a datos, lo que simplifica la trazabilidad entre requisitos, código y despliegue.

Los contenedores logicos que forman la solución se sintetizan en la Figura~\ref{fig:system-containers} y son los siguientes:

\begin{itemize}
    \item \textbf{Cliente web}: interfaz renderizada con \texttt{Next.js} y \texttt{React}, diferenciada por rol.
    \item \textbf{Servidor de aplicación}: \texttt{Route Handlers}, autenticación, protección de rutas, validación y orquestacion de servicios.
    \item \textbf{Base de datos relacional}: \texttt{PostgreSQL}, persistida y versionada mediante migraciones.
    \item \textbf{Servicios externos}: \texttt{Cloudinary} para imágenes, \texttt{Nominatim} para geocodificación, \texttt{OSRM} para rutas y \texttt{QuickChart} para la representación del QR operativo de pedido.
    \item \textbf{Capa PWA básica}: manifiesto web, iconos y \textit{service worker} ligero para soporte instalable y cache mínima.
\end{itemize}

La aplicación resuelve además varias responsabilidades transversales desde una capa común:

\begin{itemize}
    \item autenticación y sesión con \texttt{Auth.js},
    \item control de acceso por rol,
    \item compatibilidad mínima de navegador,
    \item rate limiting sobre \texttt{/api} y login,
    \item contratos de datos compartidos entre cliente y servidor.
\end{itemize}

\subsection{Diseño detallado}

La estructura del código combina una organización por capas técnicas con subdirectorios funcionales para las áreas principales del dominio. La Figura~\ref{fig:next-components} resume los componentes internos más relevantes de la aplicación \texttt{Next.js}.

\begin{itemize}
    \item \texttt{app/admin/}, \texttt{app/commercials/} y \texttt{app/clients/}: entrada funcional por rol.
    \item \texttt{app/api/}: endpoints internos segmentados por área de negocio.
    \item \texttt{app/components/}: componentes de interfaz reutilizables, incluyendo formularios, mapas, tablas y vistas especializadas de pedido o reparto.
    \item \texttt{app/hooks/api/}: capa cliente para acceso declarativo a la API.
    \item \texttt{lib/contracts/}: DTOs y contratos comunes.
    \item \texttt{lib/commercial/}: lógica de planificación de rutas y cálculo temporal reutilizable.
    \item \texttt{lib/clients/}: lectura de ETA y resumen de reparto para la zona cliente.
    \item \texttt{lib/orders/}: utilidades operativas del dominio de pedidos, como el soporte QR.
    \item \texttt{lib/geocoding/}: normalización geográfica y resolución de direcciones.
    \item \texttt{lib/security/}: políticas y utilidades de seguridad transversal.
    \item \texttt{lib/typeorm/entities/} y \texttt{lib/typeorm/services/}: modelo persistente y servicios de dominio asociados a la base de datos.
    \item \texttt{migrations/typeorm/}: evolución versionada del esquema.
\end{itemize}

\paragraph{Flujo tipico de una operación protegida}

El flujo interno de una operación sensible sigue de forma general los siguientes pasos:

\begin{enumerate}
    \item La petición llega a \texttt{proxy.ts}, donde se aplica control de navegador, autenticación base y rate limiting.
    \item La página o el endpoint correspondiente obtiene la sesión y comprueba el rol requerido.
    \item El \textit{Route Handler} delega la lógica de negocio en servicios o helpers de \texttt{lib/}.
    \item Si se requiere persistencia, la operación utiliza \texttt{TypeORM} sobre entidades y repositorios.
    \item La respuesta se normaliza y se devuelve al cliente mediante contratos compartidos.
\end{enumerate}

\paragraph{Relaciones entre módulos}

La arquitectura actual no separa los módulos como aplicaciones distintas, pero si mantiene responsabilidades de dominio claras:

\begin{itemize}
    \item \texttt{M1} aporta autenticación, roles, sesión y trazabilidad, y se reutiliza en todo el sistema.
    \item \texttt{M2} centraliza clientes, comerciales, visitas y rutas, y presta soporte logico directo a \texttt{M4}.
    \item \texttt{M3} expone el catálogo y la coloración consumidos por cliente, comercial y administrador.
    \item \texttt{M4} reutiliza visitas de \texttt{M2} para reparto y referencias de \texttt{M3} para las líneas de pedido.
    \item \texttt{M5} se apoya en el catálogo de \texttt{M3} para registrar productos y tonalidades usadas en servicios técnicos del salón.
    \item \texttt{M7}, en su estado actual, no constituye un subsistema aislado de negocio, sino una capa transversal de auditoría, configuración y soporte.
\end{itemize}

\paragraph{Organización actual por carpetas}

La arquitectura de carpetas actual es adecuada para el alcance ya construido, porque separa claramente vistas por rol, componentes reutilizables, integraciones externas, persistencia y migraciones.

No obstante, el crecimiento conjunto de los módulos comercial, catálogo y pedidos ya apunta a una evolución deseable: reforzar la agrupacion por \textit{feature} para reducir la dispersión entre componentes, hooks, contratos y servicios relacionados con una misma capacidad de negocio.
