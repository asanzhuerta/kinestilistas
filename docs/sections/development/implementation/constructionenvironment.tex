En esta sección se describe el entorno tecnologico realmente utilizado para construir la aplicación y ponerla en marcha en local.

\subsection{Marco tecnologico}

La versión actual del sistema se apoya en el siguiente conjunto de tecnologías:

\begin{itemize}
    \item \texttt{Next.js 16} como framework full-stack principal, incluyendo \textit{App Router}, \textit{Route Handlers} y estados de carga por segmento \citep{nextjsdocs2026,nextjsroutehandlers2026,nextjsloading2026}.
    \item \texttt{React 19} para la construcción de la interfaz \citep{reactdocs2026}.
    \item \texttt{TypeScript 5} como lenguaje de implementación \citep{typescriptdocs2026}.
    \item \texttt{Tailwind CSS 4} para el estilo visual \citep{tailwinddocs2026}.
    \item \texttt{Motion for React} para transiciones y microinteracciones \citep{motionreactdocs2026}.
    \item \texttt{Auth.js} para autenticación y sesión \citep{authjsdocs2026}.
    \item \texttt{TypeORM 0.3} como capa ORM y de migraciones \citep{typeormentities2026,typeormmigrations2026}.
    \item \texttt{PostgreSQL} como motor de base de datos relacional \citep{postgresdocs2026}.
    \item \texttt{Leaflet} y \texttt{react-leaflet} para visualización y selección de posiciones en mapa \citep{leafletdocs2026}.
    \item \texttt{Cloudinary} para el almacenamiento externo de imágenes \citep{cloudinaryupload2026}.
    \item \texttt{Nominatim} para geocodificación \citep{nominatimapi2026}.
    \item \texttt{OSRM} para el cálculo de rutas y geometrías \citep{osrmdocs2026}.
    \item \texttt{QuickChart} para representar el QR operativo de los pedidos \citep{quickchartqr2026}.
\end{itemize}

\subsection{Capa PWA y soporte móvil}

La aplicación expone un \texttt{manifest} web propio mediante \path{app/manifest.ts} y registra un \textit{service worker} desde \path{app/components/ServiceWorkerRegister.tsx}. En la implementación actual esto permite:

\begin{itemize}
    \item definir nombre, descripción, iconos y colores de la aplicación instalable;
    \item exponer accesos directos del manifiesto para panel, perfil, catálogo, pedidos y rutas, resueltos mediante rutas internas sensibles al rol;
    \item registrar un \textit{app shell} mínimo en \path{public/sw.js};
    \item mostrar \path{/offline} como degradación segura de navegación cuando la red falla.
\end{itemize}

No se ha construido todavía una experiencia \textit{offline-first} completa, pero la base PWA ya forma parte del código real del proyecto. De cara a la validación móvil, esta capa se beneficia de poder reutilizar una URL externa estable durante distintas sesiones de prueba. Los accesos directos añadidos al manifiesto se apoyan en \path{app/shortcuts}, que redirige a las rutas correctas de administrador, comercial o cliente sin modificar la autorización existente. El \textit{service worker} excluye rutas de API y zonas privadas del almacenamiento en cache, de forma que pedidos, rutas, perfiles y administración no se sirven como datos obsoletos. El panel \path{/admin/support} permite revisar desde administración el estado operativo de esta capa, junto con el control de navegador compatible y otras protecciones transversales.

\subsection{Estados de carga y navegación percibida}

La aplicación incorpora estados de carga por segmento mediante ficheros \path{loading.tsx} del \textit{App Router}. Esta convención permite que \texttt{Next.js} muestre una interfaz provisional mientras se resuelven datos de servidor en secciones con consultas más pesadas, como catálogo, coloración, pedidos, fichas de salón, rutas, comunicaciones, auditoría y operaciones administrativas.

Estos estados no modifican las reglas de negocio ni introducen nuevas entidades persistentes. Su finalidad es mejorar la experiencia de navegación: en lugar de esperar ante una transición vacía, el usuario visualiza una estructura esqueleto coherente con la página que se está cargando. La implementación se centraliza en \path{app/components/loading/RouteLoadingState.tsx}, reutilizado por las distintas rutas para mantener consistencia visual y evitar duplicación de marcado.

\subsection{Optimización responsive de imágenes}

Las imágenes principales del catálogo, las cartas de color y las galerías técnicas se renderizan con \texttt{next/image}, declarando tamaños responsive mediante el atributo \texttt{sizes} cuando la imagen puede ocupar anchos distintos según el dispositivo \citep{nextjsimage2026}. Esta decisión evita solicitar imágenes más grandes de lo necesario en móvil y reduce el riesgo de que las tarjetas de producto muestren recursos poco nítidos al estirar una imagen optimizada para avatar.

Para mantener la reutilización existente, el componente \path{app/components/users/UserAvatar.tsx} acepta ahora parámetros de tamaño intrínseco y \texttt{sizes}. Los avatares personales conservan tamaños pequeños, mientras que las tarjetas de producto pueden pedir una imagen de mayor resolución acorde con el espacio real que ocupan.

\subsection{Cache selectiva de datos de catálogo}

La aplicación aplica cache de datos de forma limitada sobre lecturas casi estáticas del catálogo activo y de la coloración disponible para cliente y comercial. Esta capa se implementa con la cache de datos de \texttt{Next.js}, asociando las consultas compartidas a un tag de catálogo y a una revalidación temporal corta \citep{nextjscaching2026}.

La política se mantiene deliberadamente conservadora: no se cachean pedidos, rutas, promociones activas, sesiones, perfiles ni datos privados de operación diaria. Cuando administración crea o modifica productos, líneas, subcategorías, cartas de color, referencias cromáticas o recursos de apoyo, los servicios de escritura invalidan el tag correspondiente para evitar que las vistas instaladas o navegadas desde cliente y comercial conserven información obsoleta.

\subsection{Herramientas de trabajo}

Durante el desarrollo del proyecto se han utilizado principalmente:

\begin{itemize}
    \item \texttt{Visual Studio Code} como entorno de edicion.
    \item \texttt{Git} y \texttt{GitHub} para control de versiones.
    \item \texttt{Docker Desktop} para levantar PostgreSQL en local \citep{dockerdocs2026}.
    \item \texttt{Port Forwarding} de \texttt{Visual Studio Code}, apoyado en \texttt{Microsoft Dev Tunnels}, para pruebas desde dispositivos externos \citep{vscodeportforwarding2026,microsoftdevtunnelsfaq2026}.
    \item \texttt{MiKTeX} y \texttt{latexmk} para compilar la memoria técnica.
\end{itemize}

En fases anteriores del desarrollo también se utilizo \texttt{Cloudflare Tunnel} para exponer la aplicación local \citep{cloudflaretunnel2026}. La preferencia actual por la solución integrada en \texttt{Visual Studio Code} se debe a que reduce friccion operativa, no requiere una herramienta adicional, no introduce un coste extra dentro del flujo normal de desarrollo y permite conservar URLs estables al reutilizar el mismo túnel, algo especialmente comodo para las pruebas repetidas de instalación y uso de la PWA en móvil.

\subsection{Arranque del entorno local}

La puesta en marcha del proyecto en una maquina de desarrollo sigue la secuencia mostrada a continuación:

\begin{enumerate}
    \item Clonar el repositorio.
    \item Levantar el servicio PostgreSQL definido en \texttt{docker-compose.yml}.
    \item Instalar dependencias dentro de \texttt{app-tfg/}.
    \item Configurar las variables de entorno.
    \item Ejecutar las migraciones de \texttt{TypeORM}.
    \item Iniciar el servidor de desarrollo de \texttt{Next.js}.
\end{enumerate}

Los comandos principales son los siguientes:

\begin{lstlisting}[language=bash]
docker compose up -d
cd app-tfg
npm install
npm run migration:run
npm run dev
\end{lstlisting}

\subsection{Variables de entorno}

La aplicación requiere al menos las siguientes variables:

\begin{lstlisting}[language=bash]
DATABASE_URL=postgres://kin:kin@localhost:5432/kin
AUTH_SECRET=valor-seguro
\end{lstlisting}

Existen además variables opcionales asociadas a integraciones concretas:

\begin{itemize}
    \item \texttt{CLOUDINARY\_*}: necesarias para la subida de imágenes de perfil y de catálogo.
    \item \texttt{GEOCODING\_*}: permiten personalizar el proveedor y el comportamiento de geocodificación.
    \item \texttt{NEXT\_PUBLIC\_OSRM\_BASE\_URL}: permite sustituir la URL publica por defecto del motor de rutas.
\end{itemize}

\subsection{Validación técnica disponible}

La validación automatizada actualmente consolidada en el proyecto se apoya en:

\begin{lstlisting}[language=bash]
npm run lint
npm run typecheck
npm run build
npm run m4:closeout
npm run m4:ui-check
\end{lstlisting}

Junto a estas comprobaciones se realiza verificación manual por rol, especialmente sobre login, perfil, clientes, visitas, catálogo, pedidos, reparto y cobro. La existencia de scripts específicos para el cierre de \texttt{M4} no limita su utilidad a ese módulo: complementan la comprobacion estatica general de toda la aplicación y proporcionan evidencia repetible sobre algunos de sus flujos más sensibles.
