Este capítulo describe la implementación real del sistema, centrada en la aplicación web desarrollada en \texttt{Next.js}, la capa de persistencia basada en \texttt{TypeORM} y la evolución ya materializada de los módulos \texttt{M1}, \texttt{M2}, \texttt{M3} y \texttt{M4}, junto con bloques operativos ya disponibles de \texttt{M5} y \texttt{M6}, y ciertas capacidades transversales de administración.

\section{Código Fuente}

La implementación actual del proyecto se concentra en la carpeta \texttt{app-tfg/}, mientras que la carpeta \texttt{docs/} almacena la memoria técnica en \LaTeX. Dentro de la aplicación principal se distinguen los siguientes bloques:

\begin{itemize}
    \item \texttt{app/}: páginas, layouts, \textit{Route Handlers} y componentes de interfaz.
    \item \texttt{app/api/}: endpoints internos para autenticación, perfil, administración, clientes, catálogo, pedidos, cobros y operativa comercial.
    \item \texttt{app/components/}: componentes reutilizables de interfaz, mapas, formularios, tablas y vistas por rol.
    \item \texttt{app/hooks/api/}: hooks cliente para encapsular llamadas HTTP, estado de carga y mensajes de error.
    \item \texttt{lib/}: contratos compartidos, lógica de negocio, integraciones externas, seguridad y acceso a datos.
    \item \texttt{lib/commercial/}: planificación diaria de rutas y lógica operativa del módulo comercial.
    \item \texttt{lib/clients/}: cálculo reutilizable de ETA y lectura de reparto para la zona cliente.
    \item \texttt{lib/orders/}: utilidades transversales del dominio de pedidos, entre ellas la generación y normalización de QR.
    \item \texttt{lib/security/}: políticas de rate limiting y lógica transversal de seguridad.
    \item \texttt{lib/typeorm/services/communications/}: reglas de negocio del centro de comunicaciones, promociones, formaciones, notificaciones y recordatorios.
    \item \texttt{migrations/typeorm/}: migraciones versionadas de base de datos.
\end{itemize}

\subsection{Criterios transversales de implementación}

La construcción se ha guiado por varios criterios comunes a todos los módulos, con el objetivo de evitar que cada funcionalidad evolucione de forma aislada:

\begin{itemize}
    \item mantener las páginas y componentes centrados en presentación, delegando reglas de negocio en servicios de dominio;
    \item utilizar \textit{Route Handlers} como frontera entre interfaz y servidor, con validaciones de entrada antes de llegar a persistencia;
    \item centralizar contratos y tipos compartidos para reducir discrepancias entre cliente y servidor;
    \item versionar cualquier cambio estructural de base de datos mediante migraciones trazables;
    \item reutilizar catálogos persistentes para roles, estados y tipos operativos, evitando literales dispersos en el código;
    \item aislar integraciones externas en helpers o servicios concretos para facilitar su sustitución o degradación controlada;
    \item preservar la coherencia entre roles, de modo que una misma entidad de negocio pueda ser consultada desde cliente, comercial o administrador sin duplicar reglas incompatibles.
\end{itemize}

Estos criterios permiten que el sistema siga siendo comprensible a medida que crecen los módulos y facilitan la localización de responsabilidades cuando se incorporan nuevas funcionalidades.

\subsection{Implementación del módulo M1}

El módulo \texttt{M1} se apoya principalmente en \texttt{auth.ts}, \texttt{proxy.ts}, los endpoints de \texttt{/api/auth/*}, los servicios de usuario de \texttt{lib/typeorm/services/users/} y las entidades de seguridad y trazabilidad.

Las piezas más relevantes son las siguientes:

\begin{itemize}
    \item autenticación mediante proveedor de credenciales de \texttt{Auth.js}, con sesión \texttt{JWT};
    \item control de acceso por rol en \texttt{proxy.ts}, que protege las rutas \path{/admin}, \path{/commercials} y \path{/clients};
    \item registro de solicitudes de alta y gestión administrativa de estados de cuenta;
    \item registro de eventos de acceso y trazabilidad de sesiones exitosas o fallidas;
    \item gestión y edicion del perfil propio, incluyendo cambio de contraseña y subida de imagen a \texttt{Cloudinary};
    \item limitacion de intentos de autenticación y diferenciacion explicita entre credenciales incorrectas y bloqueo temporal por rate limiting.
\end{itemize}

\subsection{Implementación del módulo M2}

El módulo \texttt{M2} se reparte entre las rutas de administración de clientes, la zona comercial, la zona cliente y la lógica de planificación diaria ubicada en \texttt{lib/commercial/}.

Actualmente ya estan implementados:

\begin{itemize}
    \item gestión de clientes profesionales y de sus datos de ficha;
    \item asignación, desasignación y reasignacion de clientes a comerciales;
    \item geolocalización de establecimientos mediante búsqueda de dirección y confirmación manual sobre mapa;
    \item configuración operativa del comercial para jornada base, puntos de salida y duración de visitas;
    \item alta, consulta y seguimiento de visitas comerciales por fecha, tipo y estado;
    \item previsualización de ruta diaria con orden previsto, tiempos estimados, margen disponible e incidencias de franja horaria;
    \item consulta por parte del cliente de la hora aproximada de reparto cuando existe una visita de entrega planificada para el día.
\end{itemize}

La lógica temporal de rutas y estimaciones no reside en la capa visual, sino que se centraliza en \path{lib/commercial/daily-route-planning.ts}. Esta decisión permite reutilizar el mismo cálculo tanto en el panel comercial como en la tarjeta de ETA que visualiza el cliente.

\subsection{Implementación del módulo M3}

El módulo \texttt{M3} se apoya en las rutas de administración bajo \texttt{/admin/catalog}, en las vistas publicas de consulta distribuidas entre las zonas cliente y comercial, y en la lógica de dominio ubicada en \texttt{lib/typeorm/services/catalog/}.

Actualmente ya estan implementados:

\begin{itemize}
    \item gestión administrativa de categorías, líneas comerciales, subcategorías jerárquicas y productos;
    \item gestión de recursos de apoyo vinculados a productos o líneas, como fichas técnicas, catálogos comerciales o materiales formativos;
    \item consulta del catálogo de productos activos por parte de clientes y comerciales, con filtros por categoría, línea comercial y subcategoría;
    \item visualización de fichas de producto con información comercial, datos técnicos, recursos asociados y enlaces a coloración relacionada cuando procede;
    \item gestión y consulta de cartas de color y referencias cromáticas, incluyendo navegacion por línea comercial y localizacion directa de tonos;
    \item subida y reemplazo de imágenes del catálogo mediante \texttt{Cloudinary}, con validación de tamano, tipo y limpieza de imágenes sustituidas.
\end{itemize}

Desde el punto de vista interno, \texttt{M3} reutiliza la misma aproximación de servicios transaccionales ya usada en módulos anteriores, pero especializada en el dominio de catálogo. Las validaciones de consistencia entre categoría, línea, subcategoría, estado y recursos asociados se centralizan en servicios internos de catálogo.

\subsection{Implementación del módulo M4}

El módulo \texttt{M4} se construye sobre las rutas de pedidos y cobros para cliente, comercial y administrador, los servicios de dominio ubicados en \path{lib/typeorm/services/orders/} y la extension del subsistema de visitas comerciales para soportar repartos reales.

Actualmente ya estan implementados:

\begin{itemize}
    \item creacion, recuperación y confirmación de borradores de pedido tanto desde cliente como desde comercial;
    \item consulta de historial y detalle de pedidos para cliente, comercial y administrador, con estados de pedido y de cobro diferenciados;
    \item regla de negocio que limita a un maximo de dos pedidos abiertos pendientes de cobro por cliente;
    \item bandeja comercial de pedidos confirmados pendientes de reparto y vinculacion de dichos pedidos a visitas de tipo \texttt{delivery};
    \item cierre de reparto con validación QR, transicion de pedidos a \texttt{delivered} y mantenimiento coherente de cancelaciones o aplazamientos;
    \item flujo mínimo de cobro mediante estados \texttt{pending}/\texttt{paid}, con registro de metodo, fecha, notas y usuario que realiza la acción;
    \item consulta de ETA para cliente solo cuando existe una visita de reparto valida y alineada con el pedido.
\end{itemize}

La lógica de negocio del módulo se concentra principalmente en \path{order.ts} y \path{commercial-visit.ts}. El helper \path{lib/orders/qr.ts} encapsula la generación y normalización del QR de cada pedido, mientras que \path{lib/clients/delivery-estimate.ts} resuelve la lectura reutilizable de la información de reparto que consume la zona cliente. Para la verificación del módulo se han incorporado además los scripts \path{m4-closeout-smoke.ts} y \path{m4-ui-headless-check.mjs}.

\subsection{Implementación del módulo M5}

El bloque actual de \texttt{M5} se apoya en las rutas \texttt{/clients/salon-clients}, \texttt{/api/clients/salon-clients/*}, \texttt{/api/clients/salon-service-templates/*} y \texttt{/api/clients/salon-service-result-images}, y en los servicios de dominio ubicados en \path{lib/typeorm/services/salon/}. A nivel persistente, este incremento cubre tanto las tablas del historial técnico del salón como el subsistema de plantillas reutilizables, la referencia cromática concreta utilizada en cada producto de coloración y la galería visual final asociada a cada servicio.

Actualmente ya estan implementados:

\begin{itemize}
    \item alta y actualización de fichas de clientes finales del salón desde el área \texttt{client};
    \item registro de servicios con fecha, tipo de tratamiento, resultado y notas generales;
    \item persistencia separada de la ficha técnica del servicio, con descripción técnica, fórmula y observaciones;
    \item guardado de plantillas reutilizables de servicio con nombre propio, datos técnicos y productos predefinidos para acelerar registros posteriores;
    \item registro estructurado de productos utilizados en cada servicio, reutilizando el catálogo activo ya existente en \texttt{M3} y enlazando la tonalidad concreta cuando el producto procede de una carta de color;
    \item carga de varias imagenes del resultado final por servicio, almacenadas en \texttt{Cloudinary} y persistidas en una tabla hija ordenada para su consulta posterior;
    \item edición y eliminación posterior de servicios ya registrados, manteniendo coherencia entre ficha técnica, productos usados y sugerencias recalculadas;
    \item reconstrucción automática de sugerencias de producto a partir del historial técnico y del uso acumulado de producto por cliente del salón;
    \item consulta integrada del historial técnico completo desde la propia ficha del cliente final, con filtros por texto, tipo de servicio y fecha;
    \item generación de borradores técnicos por servicio mediante un endpoint específico, con asunto y cuerpo editables en interfaz, copia al portapapeles y apertura opcional del cliente de correo a través de \texttt{mailto}.
\end{itemize}

Este bloque no incorpora todavía el envío real de correos técnicos a cliente final ni su integración con un proveedor externo, pero deja resuelta la base persistente, el mantenimiento del historial, la reutilización rápida de configuraciones técnicas, la documentación visual del resultado final y la preparación asistida del mensaje técnico para la operativa diaria del salón.

\subsection{Implementación del módulo M6}

El módulo \texttt{M6} se apoya en tres bloques principales:

\begin{itemize}
    \item área administrativa de comunicaciones en \path{/admin/communications};
    \item páginas de consulta para cliente y comercial en promociones, formaciones y avisos;
    \item \textit{Route Handlers} separados entre administración y consumo por usuario.
\end{itemize}

La lógica de negocio queda centralizada en \path{lib/typeorm/services/communications/communications.ts}.

Actualmente ya están implementados:

\begin{itemize}
    \item gestión administrativa de segmentos de cliente, con alta, edición, borrado, código, nombre, descripción y criterios comerciales;
    \item asignación manual de clientes profesionales a segmentos, con trazabilidad del administrador responsable y notas internas;
    \item creación, edición, cambio de estado y borrado con confirmación inline de promociones con estado \texttt{draft}, \texttt{active} o \texttt{archived}, periodo de vigencia y asociación opcional a producto, línea comercial, cliente o segmento;
    \item consulta de promociones activas por parte de clientes y comerciales, aplicando en cliente la visibilidad derivada de su asignación a segmentos;
    \item aplicación automática de descuentos porcentuales activos sobre borradores y pedidos cuando la promoción coincide con el cliente, segmento, producto o línea comercial del pedido, manteniendo el descuento y el total final como \textit{snapshot} de línea;
    \item creación, edición, publicación, cancelación, finalización y borrado con confirmación inline de formaciones con modalidad, fecha, ubicación, contenido, capacidad opcional y estado de ciclo de vida;
    \item inscripción y cancelación de inscripción a formaciones publicadas, validando en servidor la disponibilidad de plazas;
    \item generación de notificaciones internas cuando se publican promociones o formaciones relevantes, respetando en promociones el alcance global, por cliente o por segmento definido por administración;
    \item consulta y marcado de lectura de notificaciones por parte de clientes y comerciales, con indicador de avisos sin leer en sus paneles de inicio;
    \item creación y seguimiento de recordatorios personales con fecha programada y estados \texttt{pending}, \texttt{done} o \texttt{cancelled}.
\end{itemize}

El alcance actual de \texttt{M6} se mantiene centrado en comunicación interna dentro de la aplicación y en la conexión operativa con pedidos. No se realiza todavía envío real por correo electrónico, \textit{WhatsApp} u otros proveedores externos; esta integración multicanal queda documentada como evolución futura.

\subsection{Capacidades transversales de administración}

Aunque no constituyen un módulo de negocio independiente del mismo volumen que \texttt{M1} a \texttt{M4}, la aplicación ya incorpora varias capacidades transversales que deben documentarse como parte del estado real del proyecto:

\begin{itemize}
    \item vista de auditoría administrativa y de accesos desde el área \texttt{/admin};
    \item exportación \texttt{CSV} de accesos y acciones administrativas desde \path{/admin/audit}, protegida por rol de administrador;
    \item panel de operación transversal en \path{/admin/operations}, que agrega auditoría, rate limiting, integraciones y soporte técnico;
    \item exportación \texttt{CSV} del informe operativo de \texttt{M7} desde \path{/admin/operations};
    \item configuración global de políticas de rate limiting persistida en base de datos, con diagnóstico agregado de políticas activas, personalizadas y desactivadas;
    \item panel de integraciones y soporte en \path{/admin/integrations}, con inventario operativo de Cloudinary, Nominatim, OSRM y QuickChart, mostrando proveedor, estado de configuración, uso funcional y degradación prevista;
    \item panel de soporte técnico en \path{/admin/support}, con inventario operativo de compatibilidad de navegador, PWA, \textit{service worker} y formularios seguros ante hidratación;
    \item soporte PWA basico mediante manifiesto web, iconos y \textit{service worker};
    \item compatibilidad mínima de navegador con redirección a pantalla específica de no soporte.
\end{itemize}

El inventario de integraciones se calcula en \path{lib/integrations/operational-status.ts} y se expone también mediante \path{/api/admin/integrations/status}. En este primer incremento no almacena ejecuciones historicas ni sincronizaciones externas; su finalidad es reforzar la observabilidad administrativa de los servicios de soporte que ya sostienen los módulos implementados.

El inventario de soporte técnico se calcula en \path{lib/support/operational-support.ts} y se expone mediante \path{/api/admin/support/status}. A diferencia de la configuración de rate limiting, no requiere migraciones ni tablas nuevas, ya que inspecciona activos y configuraciones existentes del propio proyecto: \path{proxy.ts}, \path{app/manifest.ts}, \path{public/sw.js}, iconos PWA y componentes transversales.

La vista agregada de operación se calcula en \path{lib/admin/operational-overview.ts} y se expone mediante \path{/api/admin/operations/status}. Este servicio combina contadores de auditoría, estado de políticas de rate limiting y resúmenes de los inventarios de integración y soporte para ofrecer una lectura ejecutiva del estado de \texttt{M7}.

El informe operativo exportable se genera en \path{lib/admin/operational-report.ts} y se expone mediante \path{/api/admin/operations/report}. Su contenido transforma las secciones y métricas del resumen de operación en un fichero \texttt{CSV}, útil para conservar evidencias de revisión sin introducir una nueva entidad persistente.

La exportación de auditoría se centraliza en \path{lib/admin/audit-export.ts} y se publica mediante \path{/api/admin/audit/export}. El servicio genera bajo demanda ficheros \texttt{CSV} para accesos y acciones administrativas recientes, con límite de filas y ventana temporal configurable por query string. De este modo, el administrador puede conservar evidencias sin duplicar información ni añadir nuevas tablas al modelo físico.

El diagnóstico de rate limiting se calcula en \path{lib/admin/rate-limit-diagnostics.ts} y se expone mediante \path{/api/admin/settings/rate-limits/status}. La vista \path{/admin/settings} utiliza esta misma lectura conceptual para mostrar un resumen operativo antes del formulario de edición, diferenciando políticas activas, personalizadas y desactivadas.

\section{Scripts de Base de datos}

La persistencia se gestiona mediante migraciones de \texttt{TypeORM}. No se han empleado procedimientos almacenados ni disparadores específicos; la lógica de negocio se mantiene en servicios TypeScript, lo que simplifica su trazabilidad junto al resto del código fuente.

Además de las tablas de negocio principales, la base de datos incorpora pequeños catálogos normalizados para roles, estados y tipos operativos. Esta decisión permite que autenticación, auditoría, visitas, rutas, catálogo y pedidos trabajen contra valores controlados y coherentes con las reglas del dominio.

La serie de migraciones actual queda organizada del siguiente modo:

\begin{itemize}
    \item Migraciones \texttt{001} a \texttt{005}: tablas base del módulo \texttt{M1}, catálogos de roles y estados, usuarios, solicitudes, logs de acceso y datos semilla iniciales.
    \item Migraciones \texttt{006} a \texttt{019}: tablas y ajustes del módulo \texttt{M2}, incluyendo clientes, comerciales, asignaciones, configuración de ruta, geodatos, tipos de visita, modelo de visitas por jornada y estado aplazado.
    \item Migraciones \texttt{020} a \texttt{028}: estructura del módulo \texttt{M3}, con catálogos auxiliares, jerarquía comercial de categorías y líneas, productos, recursos de apoyo, cartas de color, referencias cromáticas, subcategorías jerárquicas y miniaturas de coloración.
    \item Migración \texttt{029}: catálogos auxiliares del módulo \texttt{M4}, con estados de pedido y de cobro.
    \item Migración \texttt{030}: tablas \texttt{orders} y \texttt{order\_lines}, incluyendo importes consolidados y trazabilidad de líneas.
    \item Migración \texttt{031}: datos semilla de estados de pedido y del flujo mínimo de cobro.
    \item Migración \texttt{032}: soporte para referencias cromáticas pedibles y persistencia de borradores.
    \item Migración \texttt{033}: vinculacion de pedidos a visitas comerciales de reparto.
    \item Migración \texttt{034}: seguimiento mínimo de cobro sobre el propio pedido, con metodo, notas, fecha y usuario responsable.
    \item Migración \texttt{035}: tabla \texttt{app\_rate\_limit\_policies} para configuración global de políticas de rate limiting desde administración.
    \item Migración \texttt{036}: base persistente del primer incremento de \texttt{M5}, con fichas de clientes del salón, servicios, ficha técnica, uso de productos y sugerencias derivadas.
    \item Migración \texttt{037}: plantillas reutilizables de servicio para \texttt{M5}, con cabecera de plantilla y productos predefinidos asociados.
    \item Migración \texttt{038}: soporte para tonalidades enlazadas a usos de producto del salón y galería de imagenes de resultado final por servicio.
    \item Migración \texttt{039}: base persistente de \texttt{M6}, con segmentos de cliente, asignaciones cliente-segmento, promociones, formaciones, inscripciones, notificaciones internas y recordatorios.
\end{itemize}

Esta estrategia permite reconstruir el esquema de forma controlada en cualquier entorno, auditar los cambios estructurales del sistema y extender el proyecto sin depender de modificaciones manuales sobre la base de datos.
