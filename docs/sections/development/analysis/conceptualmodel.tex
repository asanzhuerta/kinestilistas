A partir de los requisitos de información definidos en la \autoref{sec:InformationRequirements}, se ha elaborado el modelo conceptual de datos correspondiente. Este modelo describe las principales entidades de información que intervienen en el sistema, sus atributos y las relaciones más relevantes entre ellas.

Para mejorar la legibilidad, el modelo conceptual se presenta por módulos. En cada módulo se recogen las clases del dominio ya existentes en la implementación o, cuando el módulo no esta construido todavía, las clases previstas a nivel analitico.

\subsection{Modelo conceptual de datos M1}
% ------------------------------------------------------------------------------

En este apartado se describe el modelo conceptual correspondiente al módulo de gestión de acceso, alta y autorización de usuarios.

\subsubsection{Clase Usuario}

La clase \textit{Usuario} representa a cada persona o entidad con acceso al sistema. Sobre ella se apoyan la autenticación, la autorización por rol y la gestión del perfil propio.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador único del usuario.
      \item \textit{email}: dirección de correo electrónico utilizada para autenticación.
      \item \textit{password\_hash}: contraseña cifrada del usuario.
      \item \textit{name}: nombre completo.
      \item \textit{phone}: teléfono de contacto.
      \item \textit{company}: empresa o entidad asociada.
      \item \textit{role}: rol asignado.
      \item \textit{status}: estado de la cuenta.
      \item \textit{profile\_image\_url}: imagen de perfil.
      \item \textit{last\_login\_at}: fecha del último acceso correcto.
      \item \textit{created\_at}: fecha de alta.
      \item \textit{updated\_at}: fecha de última actualización.
\end{itemize}

\textbf{Restricciones textuales}

\begin{itemize}
      \item El email debe ser único dentro del sistema.
      \item Todo usuario debe tener un rol valido y un estado valido.
      \item Solo los usuarios activos y autorizados pueden autenticarse.
      \item Si el usuario representa a un cliente profesional, debe existir coherencia con la entidad \textit{Cliente} del módulo \texttt{M2}.
\end{itemize}

\subsubsection{Clase Rol}

La clase \textit{Rol} define los perfiles de acceso del sistema.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador del rol.
      \item \textit{code}: código interno.
      \item \textit{name}: nombre visible.
      \item \textit{description}: descripción funcional.
\end{itemize}

\subsubsection{Clase Solicitud de registro}

La clase \textit{Solicitud de registro} representa las peticiones de alta que deben ser revisadas por administración.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador de la solicitud.
      \item \textit{email}: correo del solicitante.
      \item \textit{password\_hash}: contraseña propuesta.
      \item \textit{name}, \textit{phone}, \textit{company}: datos del solicitante.
      \item \textit{requested\_role}: rol solicitado.
      \item \textit{status}: estado de la solicitud.
      \item \textit{request\_source}: origen del alta.
      \item \textit{requested\_at}, \textit{reviewed\_at}: fechas de solicitud y revisión.
      \item \textit{reviewed\_by}: usuario administrador revisor.
      \item \textit{rejection\_reason}: motivo de rechazo, cuando exista.
      \item \textit{created\_user}: usuario generado tras aprobacion, cuando corresponda.
\end{itemize}

\subsubsection{Clase Registro de administración de usuarios}

La clase \textit{Registro de administración de usuarios} mantiene la trazabilidad de cambios de estado, rol, contraseña u operaciones similares efectuadas por administración.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador del registro.
      \item \textit{target\_user}: usuario afectado.
      \item \textit{performed\_by}: usuario administrador que ejecuta la acción.
      \item \textit{action\_type}: tipo de acción.
      \item \textit{previous\_status}, \textit{new\_status}: estados anterior y nuevo.
      \item \textit{previous\_role}, \textit{new\_role}: roles anterior y nuevo.
      \item \textit{reason}, \textit{notes}: justificacion y observaciones.
      \item \textit{created\_at}: fecha del evento.
\end{itemize}

\subsubsection{Clase Registro de acceso}

La clase \textit{Registro de acceso} recoge los eventos relacionados con autenticación y sesiones.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador del registro.
      \item \textit{user}: usuario asociado, cuando exista.
      \item \textit{email\_attempted}: credencial intentada.
      \item \textit{event\_type}: tipo de evento de acceso.
      \item \textit{result}: resultado del evento.
      \item \textit{failure\_reason}: motivo del fallo, cuando exista.
      \item \textit{session\_token}: identificador de sesión.
      \item \textit{ip\_address}, \textit{user\_agent}: datos técnicos del acceso.
      \item \textit{created\_at}, \textit{revoked\_at}, \textit{expires\_at}: fechas relevantes del evento o sesión.
\end{itemize}

% ------------------------------------------------------------------------------
\subsubsection{Clases de catálogo controlado en M1}

Además de las clases principales ya descritas, la implementación actual normaliza varios valores de seguridad y autorización en tablas de catálogo independientes. Esto evita cadenas literales dispersas y refuerza la integridad del modelo.

\textbf{Clases incluidas}

\begin{itemize}
      \item \textit{Estado de usuario}: catálogo de estados permitidos para una cuenta.
      \item \textit{Estado de solicitud de registro}: catálogo del ciclo de revisión de una alta.
      \item \textit{Origen de solicitud de registro}: procedencia funcional de la peticion de alta.
      \item \textit{Tipo de acción administrativa}: clasificación de operaciones sobre cuentas.
      \item \textit{Tipo de evento de acceso}: clasificación de eventos de autenticación y sesión.
      \item \textit{Tipo de resultado de acceso}: clasificación de resultados exitosos, fallidos o revocados.
\end{itemize}

Todas estas clases comparten la misma estructura conceptual mínima: \textit{id}, \textit{code} y \textit{name}.

\subsection{Modelo conceptual de datos M2}
% ------------------------------------------------------------------------------

En este apartado se describe el modelo conceptual correspondiente al módulo de gestión comercial y clientes.

\subsubsection{Clase Cliente}

La clase \textit{Cliente} representa a cada salón profesional atendido por el distribuidor.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}, \textit{name}, \textit{contact\_name}, \textit{tax\_id}: identificación comercial del cliente.
      \item \textit{address}, \textit{city}, \textit{postal\_code}, \textit{province}: localizacion postal.
      \item \textit{linked\_user}: usuario asociado.
      \item \textit{lat}, \textit{lng}: coordenadas confirmadas.
      \item \textit{visit\_window\_start\_time}, \textit{visit\_window\_end\_time}: franja horaria valida para visitas.
      \item \textit{notes}: observaciones comerciales.
      \item \textit{created\_at}, \textit{updated\_at}: trazabilidad temporal.
\end{itemize}

\subsubsection{Clase Comercial}

La clase \textit{Comercial} complementa al usuario comercial con información operativa para cálculo de rutas y configuración de jornada.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}, \textit{user}: identificación y vinculo con usuario.
      \item \textit{employee\_code}, \textit{territory}, \textit{notes}: información interna.
      \item \textit{workday\_start\_time}, \textit{workday\_end\_time}: horario base.
      \item \textit{delivery\_visit\_duration\_minutes}, \textit{routine\_visit\_duration\_minutes}: duraciones operativas.
      \item \textit{route\_start\_address}, \textit{route\_end\_address}: puntos de salida y cierre.
      \item \textit{route\_start\_lat}, \textit{route\_start\_lng}, \textit{route\_end\_lat}, \textit{route\_end\_lng}: coordenadas operativas.
      \item \textit{return\_to\_start}: indicador de regreso al origen.
      \item \textit{created\_at}, \textit{updated\_at}: trazabilidad temporal.
\end{itemize}

\subsubsection{Clase Asignación comercial-cliente}

La clase \textit{Asignación comercial-cliente} representa la relación trazable entre un cliente y el comercial responsable en un periodo dado.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador de la asignación.
      \item \textit{client}, \textit{commercial}: entidades relacionadas.
      \item \textit{assigned\_at}, \textit{unassigned\_at}: ciclo temporal de la asignación.
      \item \textit{assigned\_by}, \textit{unassigned\_by}: responsables administrativos.
      \item \textit{notes}: observaciones de la asignación.
\end{itemize}

\subsubsection{Clase Visita comercial}

La clase \textit{Visita comercial} representa la unidad operativa básica de campo. En la implementación actual se reutiliza tanto para visitas comerciales como para repartos.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador de la visita.
      \item \textit{client}: cliente visitado.
      \item \textit{commercial}: comercial responsable.
      \item \textit{scheduled\_date}: fecha prevista.
      \item \textit{visit\_type}: tipología de la visita.
      \item \textit{status}: estado de la visita.
      \item \textit{notes}, \textit{result}: observaciones y cierre operativo.
\end{itemize}

\subsubsection{Clase Ruta comercial}

La clase \textit{Ruta comercial} representa una composicion persistida de visitas para una jornada concreta.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador de la ruta.
      \item \textit{commercial}: comercial responsable.
      \item \textit{route\_date}: fecha de la ruta.
      \item \textit{name}: descripción o nombre visible.
      \item \textit{status}: estado de la ruta.
      \item \textit{created\_at}: fecha de creacion.
\end{itemize}

\subsubsection{Clase Visita en ruta}

La clase \textit{Visita en ruta} modela el orden de las visitas dentro de una ruta persistida.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador del enlace.
      \item \textit{route}: ruta asociada.
      \item \textit{visit}: visita incluida.
      \item \textit{visit\_order}: posición de la visita dentro de la ruta.
\end{itemize}

\subsubsection{Planificación diaria derivada}

Además de las entidades persistentes, \texttt{M2} maneja una planificación diaria calculada de forma dinámica. Esta información no se persiste como entidad propia, pero conceptualmente incluye:

\begin{itemize}
      \item punto de inicio y de fin considerados;
      \item conjunto ordenado de paradas;
      \item horas estimadas de llegada y salida;
      \item tiempos de desplazamiento y espera;
      \item margen operativo restante y sobrecarga detectada.
\end{itemize}

% ------------------------------------------------------------------------------
\subsubsection{Clases de catálogo controlado en M2}

El módulo comercial también utiliza catálogos persistentes para representar estados y tipologias de trabajo. Esta decisión es especialmente importante en \texttt{M4}, ya que el reparto reutiliza la misma infraestructura de visitas de \texttt{M2}.

\textbf{Clases incluidas}

\begin{itemize}
      \item \textit{Estado de visita comercial}: clasifica visitas planificadas, aplazadas, completadas o canceladas.
      \item \textit{Tipo de visita comercial}: distingue visitas rutinarias, comerciales y de reparto.
      \item \textit{Estado de ruta comercial}: clasifica el estado operativo de cada ruta persistida.
\end{itemize}

Estas clases mantienen la estructura \textit{id}, \textit{code} y \textit{name}, y se relacionan con \textit{Visita comercial} y \textit{Ruta comercial} como catálogos de dominio.

\subsection{Modelo conceptual de datos M3}
% ------------------------------------------------------------------------------

En este apartado se describe el modelo conceptual correspondiente al módulo de catálogo, productos y coloración.

\subsubsection{Clase Producto}

La clase \textit{Producto} representa cada articulo profesional comercializado por el distribuidor.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}, \textit{name}, \textit{reference}, \textit{description}: identificación comercial del producto.
      \item \textit{category}, \textit{product\_line}, \textit{product\_subcategory}: clasificación comercial.
      \item \textit{image\_url}: imagen principal.
      \item \textit{format}, \textit{packing}: datos comerciales de presentacion, cuando existan.
      \item \textit{technical\_info}: información técnica asociada.
      \item \textit{status}: estado dentro del catálogo.
      \item \textit{base\_price}: precio base.
      \item \textit{supplier}: proveedor, cuando corresponda.
      \item \textit{created\_at}, \textit{updated\_at}: trazabilidad temporal.
\end{itemize}

\textbf{Restricciones textuales}

\begin{itemize}
      \item La referencia comercial debe ser única.
      \item Todo producto debe pertenecer a una categoría y a una línea comercial válidas.
      \item Si el producto tiene subcategoría, esta debe pertenecer a la misma línea comercial.
\end{itemize}

\subsubsection{Clase Categoría de producto}

La clase \textit{Categoría de producto} estructura el catálogo en familias generales.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}, \textit{name}, \textit{description}, \textit{display\_order}.
\end{itemize}

\subsubsection{Clase Línea comercial}

La clase \textit{Línea comercial} organiza la oferta comercial dentro de una categoría.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}, \textit{name}, \textit{description}.
      \item \textit{category}: categoría asociada.
      \item \textit{image\_url}: imagen representativa.
      \item \textit{display\_order}: orden de visualización.
\end{itemize}

\subsubsection{Clase Subcategoría de producto}

La clase \textit{Subcategoría de producto} permite introducir una jerarquía intermedia dentro de cada línea comercial.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}, \textit{name}, \textit{description}.
      \item \textit{product\_line}: línea comercial asociada.
      \item \textit{parent\_subcategory}: subcategoría padre, cuando exista.
      \item \textit{image\_url}, \textit{display\_order}.
      \item \textit{created\_at}, \textit{updated\_at}.
\end{itemize}

\subsubsection{Clase Recurso de apoyo}

La clase \textit{Recurso de apoyo} representa documentación y materiales complementarios vinculados a un producto o a una línea comercial.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}, \textit{title}, \textit{description}.
      \item \textit{resource\_type}: tipo de recurso.
      \item \textit{resource\_url}: ubicación o enlace.
      \item \textit{product}, \textit{product\_line}: contexto de asociacion.
      \item \textit{created\_at}: fecha de alta.
\end{itemize}

\subsubsection{Clase Carta de color}

La clase \textit{Carta de color} agrupa las referencias cromáticas de una línea comercial de coloración.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}, \textit{name}, \textit{description}.
      \item \textit{product\_line}: línea comercial asociada.
      \item \textit{image\_url}: imagen principal.
      \item \textit{created\_at}, \textit{updated\_at}.
\end{itemize}

\subsubsection{Clase Referencia de color}

La clase \textit{Referencia de color} representa cada tono concreto dentro de una carta de color y permite enlazarlo con un producto pedible cuando procede.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador de la referencia.
      \item \textit{color\_chart}: carta de color asociada.
      \item \textit{product}: producto vinculado, cuando la referencia sea pedible.
      \item \textit{code}, \textit{name}, \textit{description}: identificación comercial del tono.
      \item \textit{image\_url}, \textit{thumb\_image\_url}: recursos visuales.
      \item \textit{erp\_reference}: referencia ERP, cuando exista.
      \item \textit{is\_orderable}: indicador de si la referencia puede incluirse en pedido.
      \item \textit{display\_order}: orden dentro de la carta.
\end{itemize}

\subsubsection{Clases de catálogo controlado en M3}

Dentro del módulo de catálogo, la implementación actual también persiste ciertos valores de clasificación como catálogos independientes, en lugar de tratarlos como campos libres.

\textbf{Clases incluidas}

\begin{itemize}
      \item \textit{Estado de producto}: controla la disponibilidad operativa de cada articulo.

\end{itemize}

Ambas clases reutilizan igualmente la estructura \textit{id}, \textit{code} y \textit{name} y sirven como referencia para validaciones de integridad en productos y recursos.

\subsection{Modelo conceptual de datos M4}
% ------------------------------------------------------------------------------

En este apartado se describe el modelo conceptual correspondiente al módulo de pedidos, entregas y cobros.

\subsubsection{Clase Pedido}

La clase \textit{Pedido} representa la cabecera del flujo comercial de venta.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador del pedido.
      \item \textit{client}: cliente asociado.
      \item \textit{created\_by\_user}: usuario que registra el pedido.
      \item \textit{status}: estado del pedido.
      \item \textit{payment\_status}: estado de cobro.
      \item \textit{delivery\_visit}: visita de reparto vinculada, cuando exista.
      \item \textit{payment\_method}, \textit{payment\_notes}: información mínima de cobro.
      \item \textit{paid\_at}, \textit{paid\_by\_user}: trazabilidad del cobro.
      \item \textit{total\_amount}: importe total.
      \item \textit{notes}: observaciones generales.
      \item \textit{created\_at}, \textit{updated\_at}: trazabilidad temporal.
\end{itemize}

\textbf{Restricciones textuales}

\begin{itemize}
      \item El pedido debe pertenecer a un cliente existente.
      \item El estado del pedido y el estado de cobro deben pertenecer a catálogos controlados.
      \item La visita de reparto vinculada, cuando exista, debe pertenecer al mismo cliente.
      \item El sistema limita el número de pedidos abiertos pendientes de cobro por cliente.
\end{itemize}

\subsubsection{Clase Línea de pedido}

La clase \textit{Línea de pedido} representa cada producto incluido en un pedido.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador de la línea.
      \item \textit{order}: pedido asociado.
      \item \textit{product}: producto incluido.
      \item \textit{color\_reference}: referencia cromatica asociada, cuando proceda.
      \item \textit{quantity}: cantidad solicitada.
      \item \textit{unit\_price\_snapshot}: precio unitario consolidado.
      \item \textit{discount\_percentage}: descuento aplicado.
      \item \textit{line\_total}: importe total de la línea.
      \item \textit{order\_reference\_snapshot}: referencia comercial consolidada.
      \item \textit{variant\_code\_snapshot}, \textit{variant\_name\_snapshot}: instantanea de variante.
\end{itemize}

\subsubsection{Clase Estado de pedido}

La clase \textit{Estado de pedido} representa el catálogo controlado de estados del ciclo de vida del pedido.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador del estado.
      \item \textit{code}: código interno.
      \item \textit{name}: nombre visible.
\end{itemize}

\subsubsection{Clase Estado de cobro del pedido}

La clase \textit{Estado de cobro del pedido} representa el catálogo mínimo de seguimiento del pago.

\textbf{Atributos}

\begin{itemize}
      \item \textit{id}: identificador del estado.
      \item \textit{code}: código interno.
      \item \textit{name}: nombre visible.
\end{itemize}

\subsubsection{Relación con Visita comercial}

La fase logistica del pedido no se modela como una entidad independiente. En la implementación actual reutiliza la clase \textit{Visita comercial} del módulo \texttt{M2} cuando la visita tiene tipología de reparto. Esta relación permite:

\begin{itemize}
      \item agrupar pedidos confirmados en una visita de tipo \texttt{delivery};
      \item completar la entrega validando el QR de cada pedido vinculado;
      \item calcular ETA para cliente a partir de la planificación de reparto.
\end{itemize}

\subsubsection{Clase Cliente del salón}

La clase \textit{Cliente del salón} representa a los clientes finales atendidos dentro de una peluquería profesional registrada en el sistema. Su finalidad es permitir registrar información básica de identificación y mantener un historial técnico de los servicios realizados a cada cliente.

\textbf{Atributos}

\begin{itemize}

      \item \textit{id}
            Identificador único del cliente del salón.
      \item \textit{name}
            Nombre del cliente del salón.
      \item \textit{phone}
            Teléfono de contacto del cliente del salón.
      \item \textit{email}
            Correo electrónico de contacto del cliente del salón.
      \item \textit{notes}
            Observaciones generales asociadas al cliente.
      \item \textit{created\_at}
            Fecha de creación del registro.
      \item \textit{updated\_at}
            Fecha de última actualización de la información.

\end{itemize}

\subsubsection{Clase Correo técnico generado}

La clase \textit{Correo técnico generado} representa cada comunicación electrónica elaborada a partir de la ficha técnica y del historial técnico de un cliente del salón. Su finalidad es permitir generar mensajes personalizados que incluyan tratamientos realizados, productos utilizados, sugerencias de producto y otra información relevante para la comunicación profesional con el cliente.

\textbf{Atributos}

\begin{itemize}

      \item \textit{id}
            Identificador único del correo generado.
      \item \textit{salón\_client}
            Cliente del salón destinatario del correo.
      \item \textit{recipient\_email}
            Dirección de correo electrónico a la que se dirige el mensaje.
      \item \textit{subject}
            Asunto del correo generado.
      \item \textit{body}
            Contenido del correo, incluyendo la información técnica o comercial incorporada.
      \item \textit{created\_at}
            Fecha y hora de generación del correo.
      \item \textit{sent\_at}
            Fecha y hora de envío del correo, en caso de haberse realizado.

\end{itemize}

\textbf{Restricciones textuales}

\begin{itemize}

      \item \textbf{RT-EM1 — Asociación obligatoria del correo}
            Todo correo generado debe estar asociado a un cliente del salón existente.
      \item \textbf{RT-EM2 — Destinatario válido}
            Todo correo generado debe registrar una dirección de correo electrónico válida como destinatario.
      \item \textbf{RT-EM3 — Coherencia temporal del correo}
            Si el atributo \textit{sent\_at} tiene valor, este debe ser posterior o igual a la fecha de generación del correo.
      \item \textbf{RT-EM4 — Personalización previa al envío}
            El contenido del correo podrá ser revisado y personalizado por el profesional del salón antes de su envío.
\end{itemize}

\textbf{Restricciones textuales}

\begin{itemize}

      \item \textbf{RT-SC1 — Asociación obligatoria a cliente profesional}
            Todo cliente del salón debe estar asociado a un cliente profesional existente.
      \item \textbf{RT-SC2 — Nombre obligatorio}
            Todo cliente del salón debe registrar un nombre identificativo.
      \item \textbf{RT-SC3 — Coherencia temporal del registro}
            Si el atributo \textit{updated\_at} tiene valor, este debe ser posterior o igual a la fecha de creación del registro.
      \item \textbf{RT-SC4 — Correo electrónico utilizable para comunicaciones}
            Si se desea generar o enviar un correo electrónico al cliente del salón, este debe disponer de una dirección de correo electrónico registrada válida.
\end{itemize}

\subsubsection{Clase Servicio realizado}

La clase \textit{Servicio realizado} representa cada servicio o tratamiento técnico realizado a un cliente del salón. Permite documentar los trabajos efectuados y mantener un historial técnico asociado a cada cliente.

\textbf{Atributos}

\begin{itemize}

      \item \textit{id}
            Identificador único del servicio realizado.
      \item \textit{salón\_client}
            Cliente del salón al que se aplica el servicio.
      \item \textit{performed\_by}
            Usuario que registra o realiza el servicio.
      \item \textit{service\_date}
