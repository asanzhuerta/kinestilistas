En esta sección se describen los requisitos de gestión de información (datos) que el sistema debe gestionar. Estos requisitos especifican los tipos de información que el sistema debe almacenar, consultar y mantener a lo largo de su funcionamiento.

A diferencia de los requisitos funcionales, que describen los servicios que el sistema proporciona, los requisitos de información definen las estructuras de datos necesarias para soportar dichos servicios. La definición detallada de las relaciones entre estos datos y de sus restricciones de integridad se abordará posteriormente en el modelo conceptual de datos.

% ------------------------------------------------------------------------------
\subsection{RI-M1}
% ------------------------------------------------------------------------------

El modelo de información del módulo M1 se ha diseñado con el objetivo de garantizar la integridad, el control y la trazabilidad de los usuarios del sistema, incorporando estructuras específicas para la gestión de usuarios, solicitudes de registro, eventos de acceso y acciones administrativas.

\subsubsection*{Usuario}

El sistema debe almacenar información básica de cada usuario registrado en la plataforma. Esta información permite identificar al usuario, gestionar su acceso al sistema y mantener información básica de contacto.

Los datos asociados a un usuario incluyen:

\begin{itemize}
    \item Identificador del usuario.
    \item Dirección de correo electrónico.
    \item Contraseña cifrada.
    \item Nombre completo.
    \item Teléfono de contacto.
    \item Empresa o entidad a la que pertenece.
    \item Rol asignado dentro del sistema.
    \item Estado de la cuenta de usuario.
    \item Imagen de perfil.
    \item Fecha del último acceso al sistema.
    \item Fecha de creación de la cuenta.
    \item Fecha de última actualización de la cuenta.
\end{itemize}
\subsubsection*{Rol}

El sistema debe gestionar los diferentes roles que determinan los permisos y
responsabilidades de los usuarios dentro de la aplicación.

Los datos asociados a un rol incluyen:

\begin{itemize}
    \item Identificador del rol.
    \item Código del rol.
    \item Nombre del rol.
    \item Descripción del rol.
\end{itemize}

\subsubsection*{Estado de usuario}

El sistema debe gestionar un catálogo controlado de estados de usuario para representar de forma consistente la situación operativa de cada cuenta dentro del sistema.

Los datos asociados a un estado de usuario incluyen:

\begin{itemize}
    \item Identificador del estado.
    \item Código interno del estado.
    \item Nombre visible del estado.
\end{itemize}

\subsubsection*{Estado de solicitud de registro}

El sistema debe gestionar un catálogo controlado de estados de solicitud para representar el ciclo de revisión de las peticiones de alta.

Los datos asociados a un estado de solicitud incluyen:

\begin{itemize}
    \item Identificador del estado.
    \item Código interno del estado.
    \item Nombre visible del estado.
\end{itemize}

\subsubsection*{Origen de solicitud de registro}

El sistema debe gestionar un catálogo controlado de origenes de solicitud para distinguir como se ha generado cada peticion de alta.

Los datos asociados a un origen de solicitud incluyen:

\begin{itemize}
    \item Identificador del origen.
    \item Código interno del origen.
    \item Nombre visible del origen.
\end{itemize}

\subsubsection*{Solicitud de registro}

El sistema debe almacenar las solicitudes de registro realizadas por usuarios
que desean acceder a la plataforma. Estas solicitudes podrán ser revisadas por
un administrador antes de conceder o denegar el acceso.

Los datos asociados a una solicitud de registro incluyen:

\begin{itemize}
    \item Identificador de la solicitud.
    \item Correo electrónico del solicitante.
    \item Contraseña cifrada propuesta.
    \item Nombre del solicitante.
    \item Teléfono de contacto.
    \item Empresa asociada.
    \item Rol solicitado.
    \item Estado de la solicitud.
    \item Origen de la solicitud de registro.
    \item Fecha de realización de la solicitud.
    \item Fecha de revisión de la solicitud.
    \item Usuario que realizó la revisión.
    \item Motivo de rechazo (en caso de que la solicitud sea denegada).
    \item Usuario creado a partir de la solicitud, en caso de aprobación.
\end{itemize}

\subsubsection*{Registro de administración de usuarios}

El sistema debe mantener un historial de las acciones administrativas realizadas sobre las cuentas de usuario con el objetivo de disponer de trazabilidad sobre los cambios efectuados en el sistema.

Los datos asociados a cada registro de administración incluyen:

\begin{itemize}
    \item Identificador del registro.
    \item Usuario sobre el que se realiza la acción.
    \item Usuario que realiza la acción administrativa.
    \item Tipo de acción realizada.
    \item Estado previo del usuario.
    \item Nuevo estado del usuario.
    \item Rol previo del usuario.
    \item Nuevo rol del usuario.
    \item Motivo de la acción administrativa.
    \item Notas adicionales asociadas a la acción.
    \item Fecha en la que se realizó la acción.
\end{itemize}

\subsubsection*{Registro de acceso}

El sistema debe registrar los eventos relacionados con el acceso al sistema con el objetivo de mejorar la seguridad, detectar accesos indebidos y disponer
de información para auditoría.

Los datos asociados a un registro de acceso incluyen:

\begin{itemize}
    \item Identificador del registro de acceso.
    \item Usuario asociado al evento de acceso.
    \item Dirección de correo electrónico utilizada en el intento de acceso.
    \item Tipo de evento de acceso.
    \item Resultado del intento de acceso.
    \item Motivo del fallo de acceso, en caso de error.
    \item Identificador de sesión asociado al acceso.
    \item Dirección IP desde la que se realizó el acceso.
    \item Información del agente de usuario (navegador o dispositivo).
    \item Fecha y hora del evento de acceso.
    \item Fecha de revocación de la sesión, en caso de existir.
    \item Fecha de expiración de la sesión.
\end{itemize}

\subsection{RI-M2}
% ------------------------------------------------------------------------------

\subsubsection*{Cliente}

El sistema debe almacenar la información básica de cada cliente profesional atendido por el distribuidor. Esta información permite identificar cada establecimiento, registrar los datos comerciales relevantes, mantener la relación entre la peluquería como entidad de negocio y su cuenta de acceso en la plataforma y facilitar al agente comercial la consulta de los datos de contacto y localización necesarios para la planificación de visitas y rutas comerciales.

Los datos asociados a un cliente incluyen:

\begin{itemize}
    \item Identificador del cliente.
    \item Nombre del establecimiento o peluquería.
    \item Nombre de la persona de contacto.
    \item Identificador fiscal del cliente, cuando corresponda.
    \item Dirección del establecimiento.
    \item Localidad o ciudad.
    \item Código postal.
    \item Provincia o zona.
    \item Usuario asociado al cliente dentro del sistema.
    \item Latitud y longitud del establecimiento, cuando hayan sido confirmadas.
    \item Hora de inicio de la franja permitida para visitas comerciales, cuando se configure.
    \item Hora de fin de la franja permitida para visitas comerciales, cuando se configure.
    \item Observaciones o notas comerciales.
    \item Fecha de creación del registro del cliente.
    \item Fecha de última actualización de la información del cliente.
\end{itemize}

Esta información podrá ser utilizada por el sistema para mostrar al comercial los datos necesarios para localizar al cliente durante la planificación y ejecución de rutas comerciales y para calcular si la llegada estimada a dicho establecimiento se encuentra dentro de su ventana horaria válida.

\subsubsection*{Comercial}

El sistema debe almacenar la información operativa asociada a cada usuario comercial. Esta información sirve de apoyo para configurar la jornada de trabajo, el punto de inicio y fin de ruta y las duraciones base utilizadas por el algoritmo de planificación diaria.

Los datos asociados a un comercial incluyen:

\begin{itemize}
    \item Identificador del comercial.
    \item Usuario del sistema asociado al perfil comercial.
    \item Código interno de empleado, cuando corresponda.
    \item Territorio o zona comercial, cuando se utilice.
    \item Observaciones internas del perfil comercial.
    \item Hora de inicio de la jornada de trabajo.
    \item Hora de fin de la jornada de trabajo.
    \item Duración prevista de las visitas de reparto.
    \item Duración prevista de las visitas rutinarias o comerciales.
    \item Dirección y coordenadas del punto de salida configurado.
    \item Dirección y coordenadas del punto final configurado.
    \item Indicador de regreso al punto de salida al finalizar la jornada.
    \item Fecha de creación del perfil comercial.
    \item Fecha de última actualización del perfil comercial.
\end{itemize}

\subsubsection*{Asignación comercial-cliente}

El sistema debe permitir almacenar la relación entre clientes y comerciales de forma trazable, de modo que la asignación activa de un cliente pueda cambiar a lo largo del tiempo sin perder el historial de movimientos.

Los datos asociados a una asignación comercial-cliente incluyen:

\begin{itemize}
    \item Identificador de la asignación.
    \item Cliente asociado.
    \item Comercial asociado.
    \item Fecha y hora de asignación.
    \item Fecha y hora de desasignación, cuando la relación deje de estar activa.
    \item Usuario que realizó la asignación, cuando corresponda.
    \item Usuario que realizó la desasignación, cuando corresponda.
    \item Observaciones asociadas a la asignación.
\end{itemize}

\subsubsection*{Visita comercial}

El sistema debe permitir registrar las visitas comerciales realizadas por los agentes a los clientes profesionales. Estas visitas permiten documentar la actividad comercial, registrar observaciones o resultados obtenidos durante la interacción con el cliente y servir de entrada para la planificación diaria de rutas.

Los datos asociados a una visita comercial incluyen:

\begin{itemize}
    \item Identificador de la visita.
    \item Cliente visitado.
    \item Usuario comercial que realiza la visita.
    \item Fecha prevista de la visita.
    \item Tipo de visita comercial.
    \item Estado de la visita (planificada, aplazada, realizada o cancelada).
    \item Observaciones registradas durante la visita.
    \item Resultado o conclusión de la visita.
\end{itemize}

La hora concreta de atención no se almacena como un campo fijo de la visita, sino que se deriva diariamente a partir del algoritmo de planificación, la configuración del comercial y la franja horaria admitida por el cliente.

\subsubsection*{Estado de visita comercial}

El sistema debe gestionar un catálogo controlado de estados de visita comercial para representar situaciones como planificada, aplazada, completada o cancelada.

Los datos asociados a un estado de visita comercial incluyen:

\begin{itemize}
    \item Identificador del estado.
    \item Código interno del estado.
    \item Nombre visible del estado.
\end{itemize}

El sistema debe gestionar un catálogo controlado de tipos de visita comercial para distinguir entre visitas rutinarias, visitas comerciales y visitas de reparto.

Los datos asociados a un tipo de visita comercial incluyen:

\begin{itemize}
    \item Identificador del tipo.
    \item Código interno del tipo.
    \item Nombre visible del tipo.
\end{itemize}

\subsubsection*{Ruta comercial}

El sistema debe permitir organizar las rutas de trabajo de los agentes comerciales. Una ruta representa la planificación persistida de visitas que un comercial debe realizar durante una jornada o recorrido determinado y sirve de soporte para composiciones de ruta mantenidas por el sistema.

Los datos asociados a una ruta comercial incluyen:

\begin{itemize}
    \item Identificador de la ruta.
    \item Usuario comercial responsable de la ruta.
    \item Fecha de la ruta comercial.
    \item Nombre o descripción de la ruta.
    \item Estado de la ruta.
    \item Fecha de creación de la ruta.
\end{itemize}

\subsubsection*{Estado de ruta comercial}

El sistema debe gestionar un catálogo controlado de estados de ruta comercial para distinguir rutas abiertas, completadas o canceladas según la evolución de la jornada.

Los datos asociados a un estado de ruta incluyen:

\begin{itemize}
    \item Identificador del estado.
    \item Código interno del estado.
    \item Nombre visible del estado.
\end{itemize}

\subsubsection*{Visita en ruta}

El sistema debe permitir registrar las visitas que forman parte de una ruta comercial concreta. Esta información permite definir las visitas comerciales incluidas en la ruta y el orden previsto en el que deben realizarse, facilitando así la organización del recorrido comercial diario.

Los datos asociados a una visita dentro de una ruta incluyen:

\begin{itemize}
    \item Identificador del registro.
    \item Ruta comercial asociada.
    \item Visita comercial incluida en la ruta.
    \item Orden previsto de la visita dentro de la ruta.
\end{itemize}

\subsubsection*{Planificación diaria de ruta}

El sistema debe gestionar también información derivada de planificación diaria, calculada dinámicamente a partir de las visitas previstas, la configuración operativa del comercial y la geolocalización de los clientes. Esta información no tiene por qué almacenarse como entidad persistente independiente, pero sí debe poder ser consultada y presentada a los usuarios.

Los datos gestionados dentro de una planificación diaria incluyen:

\begin{itemize}
    \item Punto de inicio de la ruta utilizado para el cálculo.
    \item Punto final previsto de la jornada.
    \item Conjunto de paradas ordenadas de la ruta.
    \item Orden previsto de visita para cada cliente.
    \item Hora estimada de llegada y salida para cada parada.
    \item Tiempo aproximado de desplazamiento y espera entre paradas.
    \item Franja horaria permitida del cliente y detección de llegadas fuera de ventana.
    \item Número de visitas previstas y su tipología.
    \item Tiempo total comprometido de la jornada.
    \item Margen operativo restante y minutos de sobrecarga, cuando existan.
\end{itemize}

\subsubsection*{Visita comercial de reparto vinculada}

La implementación actual reutiliza la entidad de visita comercial del módulo \texttt{M2} para modelar la fase logistica del pedido. Cuando un pedido queda asociado a un reparto, la relación se almacena mediante una visita comercial de tipo entrega o reparto.

Los datos relevantes de esa visita vinculada incluyen:

\begin{itemize}
    \item Identificador de la visita de reparto.
    \item Cliente asociado a la visita.
    \item Comercial responsable.

En el incremento actual de \texttt{M7}, la aplicación no persiste todavía un histórico de integraciones ni ejecuciones de sincronización. El inventario operativo visible por administración se calcula a partir de la configuración real de entorno y de los proveedores utilizados por los módulos ya construidos. La persistencia de estados históricos, ejecuciones y resultados queda reservada para una evolución posterior del módulo.

De forma análoga, el inventario de soporte técnico incorporado en \path{/admin/support} no crea una entidad persistente propia. Su información se deriva de elementos ya existentes en el proyecto, como \path{proxy.ts}, \path{app/manifest.ts}, \path{public/sw.js}, los iconos PWA y componentes transversales de experiencia de usuario. Por tanto, actúa como una vista de observabilidad administrativa y no como una tabla adicional del modelo de datos.

El resumen operativo de \path{/admin/operations} sigue el mismo criterio: agrega contadores procedentes de auditoría, configuración de rate limiting, inventario de integraciones e inventario de soporte técnico. No introduce nuevas entidades persistentes, sino que ordena información ya disponible para facilitar la revisión administrativa del módulo.

La exportación \texttt{CSV} de auditoría desde \path{/admin/audit} tampoco introduce una entidad nueva. El fichero se genera bajo demanda a partir de \texttt{user\_access\_log} y \texttt{user\_management\_log}, aplicando una ventana temporal y un límite de registros para facilitar la revisión externa de accesos y acciones administrativas sin duplicar persistencia.

El diagnóstico de rate limiting expuesto en \path{/admin/settings} y en \path{/api/admin/settings/rate-limits/status} reutiliza la configuración efectiva de políticas y sus valores por defecto. Por tanto, tampoco amplía el modelo de datos: resume la tabla de sobrescrituras persistidas y el catálogo estático de políticas definido en código.

La exportación del informe operativo de \texttt{M7} desde \path{/admin/operations} sigue la misma aproximación. El fichero \texttt{CSV} se genera a partir del resumen agregado ya calculado para la vista de operación transversal, por lo que no registra nuevas filas ni introduce tablas adicionales.
